  <name>Task 1: Add srcset, REST API, and admin modal filter methods</name>
  <files>includes/class-s3mo-url-rewriter.php</files>
  <action>
Add three new hooks to register_hooks() and their corresponding methods in S3MO_URL_Rewriter:

1. SRCSET FILTER — `wp_calculate_image_srcset` (priority 10, 5 args):

filter_srcset($sources, $size_array, $image_src, $image_meta, $attachment_id) method:
- Guard: if !S3MO_Tracker::is_offloaded($attachment_id), return $sources unchanged.
- Get local upload base via wp_get_upload_dir()['baseurl'].
- Get CDN uploads base via $this->get_cdn_uploads_base().
- Loop each $sources entry. Each entry has a 'url' key.
- str_replace the local upload base with CDN uploads base in each source URL.
- Handle both http:// and https:// variants (same approach as filter_content).
- Return modified $sources array.
- Important: wp_get_attachment_url filter fires before srcset calculation, so the src domain already matches CDN. If srcset URLs do NOT also point to CDN, WordPress drops the entire srcset. This filter prevents that.

2. REST API FILTER — `rest_prepare_attachment` (priority 10, 3 args):

filter_rest_attachment($response, $attachment, $request) method:
- $attachment_id = $attachment->ID.
- Guard: if !S3MO_Tracker::is_offloaded($attachment_id), return $response unchanged.
- Get S3 key via S3MO_Tracker::get_s3_key($attachment_id).
- Rewrite source_url in $response->data: $response->data['source_url'] = $this->client->get_object_url($key).
- Rewrite media_details.sizes: if $response->data['media_details']['sizes'] exists, loop each size entry. For each size, str_replace the local upload base with CDN uploads base in the 'source_url' field. Handle http/https variants.
- Return $response.
- This is critical for the headless Next.js frontend which reads attachment data from /wp-json/wp/v2/media.

3. ADMIN MODAL FILTER — `wp_prepare_attachment_for_js` (priority 10, 3 args):

filter_attachment_for_js($response, $attachment, $meta) method:
- $attachment_id = $attachment->ID.
- Guard: if !S3MO_Tracker::is_offloaded($attachment_id), return $response unchanged.
- Get local upload base via wp_get_upload_dir()['baseurl'].
- Get CDN uploads base via $this->get_cdn_uploads_base().
- Rewrite $response['url'] with str_replace.
- If $response['sizes'] exists, loop each size and str_replace the 'url' field.
- If $response['icon'] exists, str_replace it too.
- Handle http/https variants.
- Return $response.

Hook registration order in register_hooks():
1. wp_get_attachment_url (existing from Plan 01)
2. wp_calculate_image_srcset (new)
3. the_content (existing from Plan 01)
4. rest_prepare_attachment (new)
5. wp_prepare_attachment_for_js (new)

Important: Do NOT modify the existing filter_attachment_url or filter_content methods from Plan 01. Only ADD the new hooks and methods.

<task type="auto">
  <name>Task 2: Add CORS headers for cross-origin media requests</name>
  <files>includes/class-s3mo-url-rewriter.php</files>
  <action>
Add CORS header support to S3MO_URL_Rewriter for cross-origin requests from the Next.js frontend.

In register_hooks(), add a new hook:
- `send_headers` action at priority 10.

add_cors_headers() method:
- Only fire on REST API requests: check if defined('REST_REQUEST') && REST_REQUEST is true.
- Get the Origin header from $_SERVER['HTTP_ORIGIN'] if set.
- Define allowed origins array. Read the site URL via get_site_url() and also include common localhost development URLs: 'http://localhost:3000', 'http://localhost:3001'. If S3MO_CDN_URL is defined, include it as well.
- If Origin is in the allowed list, set these headers:
  - header('Access-Control-Allow-Origin: ' . $origin);
  - header('Access-Control-Allow-Methods: GET, HEAD, OPTIONS');
  - header('Access-Control-Allow-Headers: Content-Type, Authorization');
  - header('Access-Control-Expose-Headers: Content-Length, Content-Range');
- If no Origin match, do nothing (don't set wildcard).
- Use header() calls, not WordPress header functions (this fires on send_headers action).

Important: This is a server-side hint for the WP REST API. The actual CloudFront CORS configuration must be done in AWS (S3 bucket CORS policy + CloudFront Origin Request Policy forwarding the Origin header). Document this in a comment at the top of the method: "CloudFront CORS requires separate AWS configuration: S3 bucket CORS rules + CloudFront Origin Request Policy to forward the Origin header."

Also add a note about the CORS architecture:
- WP REST API CORS: handled by this method (for /wp-json/ requests)
- CloudFront CORS: handled by AWS config (S3 CORS + CF response headers policy)
- Both are needed for a fully working headless Next.js setup
