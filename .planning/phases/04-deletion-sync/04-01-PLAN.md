<task type="auto">
  <name>Task 1: Create S3MO_Delete_Handler class</name>
  <files>includes/class-s3mo-delete-handler.php</files>
  <action>
Create `includes/class-s3mo-delete-handler.php` following the exact same class structure as S3MO_Upload_Handler:

1. File header: `defined('ABSPATH') || exit;` guard, docblock matching upload handler style.

2. Class `S3MO_Delete_Handler` with:
   - Private property `S3MO_Client $client`
   - Constructor accepting `S3MO_Client $client` (dependency injection, same as upload handler)
   - `register_hooks()` method: `add_action('delete_attachment', [$this, 'handle_delete'], 10, 2);`
     - Priority 10 is correct: `delete_attachment` fires BEFORE postmeta/files are removed
     - Second parameter ($post) available with `2` as accepted_args

3. `handle_delete(int $post_id, \WP_Post $post): void` method:
   - Guard: `if (!S3MO_Tracker::is_offloaded($post_id)) { return; }` — skip non-offloaded attachments
   - Call `$this->collect_s3_keys($post_id)` to gather all keys
   - Deduplicate with `array_unique()` (WordPress may reuse same physical file for multiple sizes)
   - Loop through keys, call `$this->client->delete_object($key)` for each
   - On failure: `error_log('CT S3 Offloader: Failed to delete S3 object: ' . $key . ' - ' . ($result['error'] ?? 'Unknown error'));`
   - Do NOT throw exceptions or return early on failure (DEL-04)
   - After loop: `S3MO_Tracker::clear_offload_status($post_id);`
   - Clear tracker AFTER S3 deletions (need the key to delete from S3 first)

4. `private function collect_s3_keys(int $post_id): array` method:
   - Get original key: `S3MO_Tracker::get_s3_key($post_id)` — return empty array if empty
   - Add original key to `$keys[]`
   - Get attachment metadata: `wp_get_attachment_metadata($post_id)`
   - If `$metadata['sizes']` exists and is array, derive thumbnail keys:
     - `$s3_dir = dirname($s3_key);`
     - For each size: `$keys[] = $s3_dir . '/' . $size_data['file'];`
   - This mirrors exactly how upload handler builds thumbnail paths (see lines 74-84 of class-s3mo-upload-handler.php), but uses the S3 key directory instead of upload_dir
   - Return `$keys`

Important: Do NOT use `wp_get_attachment_metadata()` for the original key — use `S3MO_Tracker::get_s3_key()` which stores the FULL S3 key including the path prefix (e.g., `wp-content/uploads/2026/02/image.jpg`). The metadata `file` field does NOT include the path prefix.
