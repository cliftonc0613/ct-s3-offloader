Create the upload handler that intercepts WordPress media uploads and offloads files to S3, then modify the plugin bootstrap to register it for all contexts (not just admin).

Purpose: This is the core Phase 2 deliverable. When a user uploads media through the Media Library, REST API, or any other WordPress upload mechanism, the handler intercepts the generated metadata, uploads the original file plus all thumbnail sizes to S3, and tracks the result in postmeta.

Output: New `class-s3mo-upload-handler.php` and modified `ct-s3-offloader.php` bootstrap.

<task type="auto">
  <name>Task 1: Create S3MO_Upload_Handler class</name>
  <files>includes/class-s3mo-upload-handler.php</files>
  <action>
Create `includes/class-s3mo-upload-handler.php` with class S3MO_Upload_Handler.

**Constructor: __construct(S3MO_Client $client)**
- Store $client as private property
- Do NOT register hooks in constructor (register_hooks() is separate for testability)

**Method: register_hooks(): void**
- `add_filter('wp_generate_attachment_metadata', [$this, 'handle_upload'], 10, 3)`

**Method: handle_upload(array $metadata, int $attachment_id, string $context): array**

This is the core method. It MUST always return $metadata unchanged (it's a filter, not an action).

Step 1: Guard clauses
- If `$context !== 'create'`, return $metadata immediately (only handle new uploads in Phase 2)
- If `empty($metadata['file'])`, return $metadata (no file path = nothing to upload)
- If `S3MO_Tracker::is_offloaded($attachment_id)`, return $metadata (already done)

Step 2: Build upload list
- Get upload dir: `$upload_dir = wp_get_upload_dir()` — use `$upload_dir['basedir']` for local path
- Get path prefix: `$prefix = get_option('s3mo_path_prefix', 'wp-content/uploads')`
- Original file: local path = `$upload_dir['basedir'] . '/' . $metadata['file']`, S3 key = `$prefix . '/' . $metadata['file']`
- Get original MIME: `$mime = get_post_mime_type($attachment_id)`
- Build array of files to upload: `$files = [['local' => $local_path, 'key' => $s3_key, 'mime' => $mime]]`
- For each `$metadata['sizes']` entry: local path = `$upload_dir['basedir'] . '/' . dirname($metadata['file']) . '/' . $size_data['file']`, S3 key = `$prefix . '/' . dirname($metadata['file']) . '/' . $size_data['file']`, MIME = `$size_data['mime-type']`
- Add each thumbnail to the $files array

CRITICAL: Thumbnail `$size_data['file']` is FILENAME ONLY (e.g., 'photo-150x150.jpg'). The directory comes from `dirname($metadata['file'])` which provides the relative year/month path (e.g., '2026/02').

Step 3: Upload each file independently
- Initialize `$errors = []` and `$success_count = 0`
- Loop through $files array. For EACH file individually (not batch):
  - Wrap in try/catch
  - Check `file_exists($file['local'])` — if not, log error and continue (do not fail)
  - Call `$result = $this->client->upload_object($file['key'], $file['local'], $file['mime'])`
  - If `$result['success']`: increment $success_count
  - If not: add to $errors array, log with `error_log('CT S3 Offloader: Failed to upload ' . $file['key'] . ' — ' . $result['error'])`
  - Catch any Exception: log it, add to $errors, continue to next file

CRITICAL: Each file upload MUST be in its own try/catch. A failed thumbnail must NOT prevent other thumbnails from uploading. This satisfies UPLD-05.

Step 4: Track result
- `$total = count($files)`
- If `$success_count === $total` (ALL succeeded): call `S3MO_Tracker::mark_as_offloaded($attachment_id, $files[0]['key'], $this->client->get_bucket())`
- If `$success_count > 0 && $success_count < $total`: log partial failure with count, do NOT mark as offloaded
- If `$success_count === 0`: log complete failure, do NOT mark as offloaded

Only mark as offloaded when ALL files succeed. The S3 key stored is the ORIGINAL file's key (index 0).

Step 5: Return $metadata unchanged (ALWAYS — this is a filter, breaking the return breaks WordPress)

Add PHPDoc, `defined('ABSPATH') || exit;` guard. No use statements needed (S3MO_ classes autoload, no namespaced dependencies).

<task type="auto">
  <name>Task 2: Modify bootstrap to register upload handler for all contexts</name>
  <files>ct-s3-offloader.php</files>
  <action>
Modify the plugins_loaded callback in ct-s3-offloader.php. Currently, ALL component instantiation is inside `if (is_admin())`. The upload handler MUST be registered OUTSIDE is_admin() because uploads happen via REST API (Gutenberg editor, headless frontend).

Change the plugins_loaded callback structure from:

```php
if (is_admin()) {
    $client = empty($missing) ? new S3MO_Client() : null;
    $settings = new S3MO_Settings_Page($client);
    $settings->register_hooks();
}
```

To:

```php
// Upload handler must run in ALL contexts (admin, REST API, cron)
// because Gutenberg and headless frontends upload via REST API
if (empty($missing)) {
    $client = new S3MO_Client();

    $upload_handler = new S3MO_Upload_Handler($client);
    $upload_handler->register_hooks();

    if (is_admin()) {
        $settings = new S3MO_Settings_Page($client);
        $settings->register_hooks();
    }
} elseif (is_admin()) {
    // Settings page still works with null client (shows connection error)
    $settings = new S3MO_Settings_Page(null);
    $settings->register_hooks();
}
```

Key changes:
1. S3MO_Client is created OUTSIDE is_admin() when credentials are present
2. S3MO_Upload_Handler is registered OUTSIDE is_admin()
3. Settings page remains admin-only
4. When credentials are missing: settings page still loads (with null client) but upload handler is NOT registered (no client = can't upload)
5. $client is shared between upload handler and settings page (one instance)

Do NOT change anything else in the file — activation hooks, autoloader, SDK loading, constants must remain exactly as they are.
  </action>
  <verify>
Open ct-s3-offloader.php and confirm:
1. S3MO_Upload_Handler is instantiated OUTSIDE `is_admin()`
2. S3MO_Upload_Handler->register_hooks() is called
3. S3MO_Settings_Page is still inside `is_admin()`
4. When credentials are missing, upload handler is NOT created
5. Settings page still gets null client when credentials are missing
6. `php -l ct-s3-offloader.php` passes
  </verify>
  <done>
Bootstrap registers upload handler for all contexts (admin, REST API, cron). Settings page remains admin-only. Single S3MO_Client instance shared between components. Missing credentials gracefully skip upload handler while still showing settings page.
  </done>

  <how-to-verify>
1. Ensure wp-config.php has all four constants defined: S3MO_BUCKET, S3MO_REGION, S3MO_KEY, S3MO_SECRET
2. Go to WordPress Admin > Media > Add New Media
3. Upload a JPEG image (this generates multiple thumbnail sizes)
4. Check S3 bucket — the original and thumbnail sizes should appear under the configured path prefix (e.g., wp-content/uploads/2026/02/filename.jpg and wp-content/uploads/2026/02/filename-150x150.jpg, etc.)
5. In S3 console, click on one of the uploaded objects and check:
   - Content-Type should be `image/jpeg`
   - Cache-Control should be `public, max-age=31536000, immutable`
   - ACL should NOT be public-read (object is private, served via CloudFront OAC)
6. In WordPress, check the attachment postmeta (via database or a postmeta viewer plugin):
   - `_s3mo_offloaded` should be `1`
   - `_s3mo_key` should contain the S3 key path
   - `_s3mo_bucket` should match your bucket name
   - `_s3mo_offloaded_at` should have a recent ISO 8601 timestamp
7. Check `wp-content/debug.log` — there should be NO CT S3 Offloader error entries for this upload
